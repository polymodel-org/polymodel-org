03 — Core concepts

A small, honest vocabulary.

Entities carry identity. Components are reusable field groups. Relationships associate entities. Projections map the logical model to a target.

Entity

An identity-bearing business object — User, Order, Invoice.

entity "User" {
  field "id"    { type = uuid }
  field "email" {
    type   = string
    unique = true
  }
}

Component

A reusable group of fields with no independent identity. Composition over inheritance.

component "Auditable" {
  field "createdAt" { type = timestamp }
  field "updatedAt" { type = timestamp }
}

Relationship

A logical association between entities — parents, children, hierarchies.

entity "Order" {
  parent = entity.User
  field "id"        { type = uuid }
  field "createdAt" { type = timestamp }
}

Composition

Pull components into entities — much like Go struct embedding.

entity "Invoice" {
  use component.Auditable
  field "total" {
    use component.CurrencyAmount
  }
}

04 — One model, many shapes

Hierarchy is a logical concern, not a storage one.

The same User → Order → OrderItem model projects into nested documents, flat relational tables, or rich DTOs — without touching the source. 

entity "User"      { field "id" { type = uuid } }
entity "Order"     { parent = entity.User  field "createdAt" { type = timestamp } }
entity "OrderItem" { parent = entity.Order field "quantity"  { type = int } }

05 — Cross-repo composition

Import models the way Go imports packages.

Models don't live in one file. Reference a published module by its repository path; the last path segment becomes a namespace you compose with — pinned to immutable version tags through the Catalog. 

// pull in a published module by its repo path
import "github.com/polymodel-org/currency"

entity "Order" {
  field "id"     { type = uuid }
  field "total"  { use currency.Amount }
}
  • 01

    Namespace from the path

    The last segment — currency — becomes the namespace you reference. No registry lookups, no magic: the path is the identity.

  • 02

    Alias before the path

    Go-style aliasing resolves collisions and sharpens intent: import fx "…/currency" → fx.Amount.

  • 03

    Immutable, pinned versions

    Imports resolve to semver Git tags through the Catalog. The repository stays the source of truth — exactly like Go modules.

Group your imports

Collect dependencies in one block. Alias any of them inline.

import (
  "github.com/polymodel-org/currency"
  geo "github.com/polymodel-org/country"
)

Compose components & entities

Use remote components, or reference a remote entity as a field type.

entity "Customer" {
  use currency.Auditable
  field "country" {
    type = geo.Country
  }
}

Two files: your intent, and the resolved truth

You hand-author polymodel.mod in HCL — the same grammar as your models — declaring your module's identity and the versions it depends on. A resolver generates polymodel.lock: the exact commit and content checksum per module, so any machine rebuilds the same schema. The split mirrors go.mod and go.sum.

polymodel.mod hand-authored 
module "github.com/acme/orders"

polymodel = "0.1"

require "github.com/polymodel-org/currency" { version = "v1.2.0" }

# local dev override — co-located, never a stray directive elsewhere
require "github.com/polymodel-org/country" {
  version = "v0.4.0"
  replace = "../country"
}
polymodel.lock generated 
# resolved & checksum-verified via the Catalog
github.com/polymodel-org/currency  v1.2.0  commit:9f3a1c7  h1:sha256-Wqf3…aQ=
github.com/polymodel-org/country   v0.4.0  → ../country  (local replace)

Need a local checkout while developing? replace rides on the require itself — co-located and impossible to miss, unlike go.mod's free-floating replace directives — and the lock reflects it. The commit pins what resolved; the content hash proves integrity independent of git. See the Catalog →

06 — The Catalog

An open ecosystem for sharing and linking models.

A model should be definable once and reusable everywhere. The PolyModel Catalog is a Git-native registry — repositories stay the source of truth.

Git-native, like Go modules

Modules are published as Git repositories, versioned with immutable semver tags.

# Git repo + version tags = a published module
github.com/polymodel-org/currency  v1.0.0
github.com/polymodel-org/country   v0.2.0
github.com/acme/payments           v0.1.0

Canonical entities

Reusable reference entities become shared semantics across organizations.

  • Currency
  • Country
  • Address
  • Person
  • Company
  • ExchangeRate

Automatic dataset linking

When datasets map to the same canonical entities, tooling can infer joins, enrichments, and navigation — no shared storage required.

Orders.currency_code   ↕   ExchangeRates.currency_code
Orders.order_date      ↕   ExchangeRates.date

The network effect comes from shared semantics, not shared storage.

07 — Roadmap

From specification to stable ecosystem.

PolyModel is in the design phase. The spec is evolving and may change before v1.0.

  1. v0.1

    Foundations

    Entities, fields, relationships, hierarchies, HCL syntax, the specification document, example models.

  2. v0.2

    First generators

    PostgreSQL, SQLite, Go, and TypeScript generators.

  3. v0.3

    Documents & transport

    Firestore & MongoDB projections, JSON Schema and OpenAPI generators.

  4. v0.4

    Evolution

    Migration specification, index definitions, constraints, validation rules.

  5. v0.5

    Tooling

    GraphQL & TypeSpec generators, VS Code extension, formatter.

  6. v1.0

    Stable

    Stable spec, compliance suite, reference implementations, community governance.

What PolyModel is not

PolyModel defines the specification. Implementations build on top of it.

  • An ORM
  • A database
  • An API framework
  • A migration engine
  • A code generator implementation

Define the model once.
Let everything else follow.

PolyModel is an open specification. Feedback, ideas, and contributions are welcome.
